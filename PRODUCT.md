# Lockr

Lockr is a self-hosted secrets manager written in Go. It runs as a single binary with embedded BadgerDB storage, encrypted secret values, service authentication, policy checks, and audit logging.

The current project is backend/CLI focused. There is no web UI and no external storage backend.

## What It Stores

Lockr currently supports three secret types:

- KV secrets: JSON values stored by path, encrypted at rest, with up to five retained versions.
- Transit keys: named AES keys used to encrypt/decrypt caller-provided plaintext without exposing key material.
- Dynamic Postgres credentials: short-lived database users created from stored admin connection config.

KV deletes are soft deletes in metadata. A restore/purge API is not currently implemented.

## Security Model

Transport:

- Production mode uses HTTPS with TLS 1.3.
- `lockr init` creates a local CA and server certificate.
- Clients use `ca.crt` to verify the Lockr server.
- Mutual TLS client certificate authentication is not implemented.

Encryption at rest:

- Secret records are encrypted before writing to BadgerDB.
- The master key is generated by `lockr init` and stored encrypted in `master.key.enc`.
- The server unlocks the master key using `LOCKR_PASSPHRASE`.
- Per-record keys are derived from the master key using HKDF-SHA256.

Authentication:

- Ed25519 challenge-response is the main service auth flow.
- Admin tokens are long-lived operator/automation tokens.
- TOTP support exists for services that cannot use Ed25519.
- Dev mode bypasses auth and grants root policy; use it only for local testing.

Session tokens:

- Successful service auth returns an `lvt_...` session token.
- Sessions store a deterministic token lookup key plus an Argon2id verification hash.
- The plaintext session token is not stored.
- Default session TTL is one hour.

## Ed25519 Flow

1. An admin enrolls a service.
2. Lockr generates an Ed25519 keypair.
3. Lockr stores the public key at `auth/services/<service>`.
4. The private key is returned once and saved by the service.
5. The service requests `/v1/auth/challenge`.
6. The service signs the challenge with its private key.
7. Lockr verifies the signature with the stored public key.
8. Lockr issues a short-lived session token.
9. The service uses the session token for normal API requests.

The private key is only used to authenticate. It is not sent to Lockr and is not used directly to read secrets.

## Policies

Policies are YAML files loaded from the configured policy directory:

```yaml
policy:
  dir: "/etc/lockr/policies"
```

Example:

```yaml
name: api-server

rules:
  - path: "secrets/kv/prod/api/*"
    capabilities: [read]

  - path: "secrets/transit/payments-key"
    capabilities: [encrypt, decrypt]
```

Capabilities:

- `read`
- `write`
- `delete`
- `list`
- `encrypt`
- `decrypt`

Path matching supports exact paths and a single trailing wildcard such as `secrets/kv/prod/api/*`. Explicit deny rules override allow rules. If no rule matches, access is denied.

Policies are loaded on startup and can be reloaded with:

```bash
lockr policy reload --token <admin-token>
```

or by sending `SIGHUP` to the server process.

## BadgerDB Structure

BadgerDB is a key-value database. Lockr creates structure with key prefixes:

```text
auth/services/<name>           Ed25519 service public key and policy
auth/services/totp/<name>      TOTP service record
auth/admins/<name>             Admin token hash and metadata
auth/sessions/<token-lookup>   Session metadata

secrets/kv/<path>/__meta       KV metadata
secrets/kv/<path>/v<N>         KV version record
secrets/transit/<key>          Transit key metadata and key versions
secrets/db/<name>              Dynamic Postgres config
secrets/db/leases/<lease-id>   Active dynamic credential lease

audit/<timestamp>/<ulid>       Queryable audit entry
```

Policies are file-backed in the current implementation. They are not stored in BadgerDB.


## Local Dev Run

Build:

```bash
go build -o lockr ./cmd/lockr
```

Start dev mode:

```bash
./lockr server --dev
```

Dev mode uses HTTP, in-memory storage, and no auth enforcement.

Create an admin token in dev mode:

```bash
./lockr admin create --name root --addr http://localhost:8300
```

Enroll a service:

```bash
./lockr enroll \
  --service api-server \
  --auth ed25519 \
  --policy root \
  --output ./certs \
  --addr http://localhost:8300 \
  --token <admin-token>
```

Use service auth from the CLI:

```bash
./lockr get secrets/prod/api/db \
  --addr http://localhost:8300 \
  --identity api-server \
  --key ./certs/api-server.key
```

For dev/admin-style direct token usage:

```bash
./lockr set secrets/prod/api/db '{"password":"secret"}' \
  --addr http://localhost:8300 \
  --token <admin-token>
```

## Production-Style Run

Initialize once:

```bash
lockr init --data-dir /var/lib/lockr
```

Start:

```bash
LOCKR_PASSPHRASE=<passphrase> lockr server --config /etc/lockr/config.yml
```

In another terminal, verify health:

```bash
curl -k https://localhost:8300/v1/sys/health
```

Create the first admin token. This is allowed only while no admin tokens exist:

```bash
lockr admin create --name root \
  --addr https://localhost:8300 \
  --ca /etc/lockr/tls/ca.crt
```

Save the returned `lkat_...` token. Later admin creation requires an existing admin token:

```bash
lockr admin create --name ops \
  --addr https://localhost:8300 \
  --ca /etc/lockr/tls/ca.crt \
  --token <root-admin-token>
```

Use `config.example.yml` as the current config reference.

## Current Gaps

These are known implementation gaps and should not be documented as finished behavior:

- Some sensitive DB/transit routes still need stricter policy enforcement.
- Admin and service operations should be separated more clearly.
- CLI TLS currently allows insecure verification when no CA is provided.
- mTLS client certificate auth is not implemented.
- KV restore/purge is not implemented.
- Master-key canary verification is not implemented.
- Test coverage is still limited.
